Conceitos

Cinco ideias que explicam por que o homúnculo se comporta como se comporta — e que tornam o editor e o simulador fáceis de raciocinar.

1. O que é uma árvore de comportamento

Uma árvore de comportamento (BT) é uma forma de descrever decisões como uma hierarquia de blocos. A cada instante do jogo (um tick), a árvore é percorrida do topo (raiz) para baixo, e cada bloco diz se conseguiu resolver a situação ou não. O resultado é uma única decisão: o que o homúnculo deve fazer agora.

O ponto-chave é que a árvore é reavaliada inteira a cada tick. Ela não “fica presa” num estado: se a situação muda (o dono se afasta, um monstro aparece, o HP cai), na próxima avaliação a árvore já reage. Isso elimina toda uma classe de bugs de “a IA travou num modo”.

2. A árvore só decide — ela não age

Cada tick, a árvore lê o estado do mundo e escreve uma única intenção: “mover para ali”, “atacar aquele alvo”, “usar a skill tal”. Quem executa de fato a intenção é a camada do jogo, depois. A árvore nunca chama o jogo diretamente.

Na prática, isso significa que o homúnculo faz uma coisa por tick, e essa coisa é sempre a de maior prioridade que se aplica naquele instante. É por isso que, no simulador, o painel intent do blackboard mostra exatamente o que a árvore decidiu — e o mapa mostra aquilo acontecendo.

3. Como ler os status (SUCCESS / FAILURE / RUNNING)

Todo bloco devolve um destes três resultados a cada avaliação. Entendê-los é o que torna a árvore colorida do simulador legível:

  • SUCCESS (verde) — o bloco resolveu a situação neste tick.
  • RUNNING (amarelo) — o bloco começou algo que leva mais de um tick (ex.: perseguir, conjurar) e continua no próximo.
  • FAILURE (cinza) — o bloco não se aplica agora, ou uma condição deu falso. A árvore segue para a próxima opção.

Uma ação que falhou ganha fundo vermelho no painel — é um sinal de “tentei e não consegui” (ex.: usar uma skill sem SP). Já o cinza é só “não era a vez deste ramo”.

4. Prioridade é a ordem dos filhos

O bloco mais comum é o selector: ele tenta os filhos na ordem em que estão e fica com o primeiro que dá certo. Então a ordem é a prioridade. Numa árvore típica, a raiz é um selector com mais ou menos esta ordem:

  1. obedecer ao comando do dono;
  2. sobreviver (fugir com HP baixo);
  3. curar (a si ou ao dono);
  4. engajar inimigos;
  5. ficar perto do dono / ocioso.

Como tudo é reavaliado a cada tick, basta a situação mais importante deixar de se aplicar para a árvore “descer” para a próxima — e voltar a subir assim que ela reaparecer. Reordenar os filhos no editor muda a prioridade; é uma das edições mais comuns. O outro bloco básico é o sequence (um “E”: só segue se o anterior teve sucesso). Detalhes de todos os blocos: Referência de nós.

5. A mesma árvore roda no jogo e no simulador

O simulador não é uma imitação da IA: ele roda exatamente o mesmo motor que roda dentro do cliente do RO. A única diferença é de onde vêm os “sentidos” e para onde vão as “ações” — no jogo, do cliente real; no simulador, de um mundo de mentira. A árvore não sabe a diferença.

É por isso que vale a pena testar no simulador antes de levar ao jogo: o que você vê ali é, com altíssima fidelidade, o que vai acontecer no RO. (Alguns detalhes finos do servidor real — lag, timing de cast — ainda pedem um teste final no jogo; veja a última seção do Guia do Simulador.)

✓ Por isso o pacote gerado é confiável

Quando você clica Gerar Lua, o pacote leva esse mesmo motor mais a sua árvore. Nenhuma “tradução” acontece no caminho — o que foi testado é o que roda. Nós desativados são simplesmente removidos do pacote (e por isso somem do simulador também).
